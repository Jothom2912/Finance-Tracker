# P3-14: in-flight sync-claim

Plan: [plans/2026-07-17-p314-serialize-bank-sync-sagas.md](../plans/2026-07-17-p314-serialize-bank-sync-sagas.md).
One code commit (`f1d36d22`) + docs; same-day (third item this session after P3-16 + F1-07).

## Done

- **Design pivot vs. audit M8**: NOT a deterministic correlation-id (globally unique in
  `saga_instances` — a terminal saga would block all future syncs) but an in-flight
  claim on `bank_connections` (`sync_saga_id` + `sync_started_at`, migration 003).
- `start_sync_saga`: atomic claim + outbox start-event in ONE transaction; returns
  `(saga_id, already_running)`. Conflict → status-check via saga-service (caller's own
  JWT forwarded — status API is owner-checked per P1-04): terminal → steal (UPDATE
  scoped to old saga_id, one winner), active/unknown → hand back the running saga_id.
  Unknown fails ACTIVE; `SYNC_CLAIM_TTL_SECONDS=600` (2× saga timeout) is the backstop.
- `mark_sync_complete` releases the claim only when it still belongs to the completing
  saga — an old duplicate reply can't clear a newer claim.
- API: 202 gains `already_running` (additive; frontend untouched).
- 42 tests + lint green: 6 claim-flow service tests (active/terminal/unknown/vanished/
  lost-steal-race/no-port), 2 consumer clear-claim tests.

## Live e2e (real EB-sandbox connection): PASSED

1. Two concurrent `POST /sync` → **same saga_id** (`started` + `already_running`).
2. Saga completed → claim columns cleared, `last_synced_at` bumped.
3. Third sync → **new saga_id**; claim held the new id during the run, cleared after.

## Learned / gotchas

- **banking-service has no venv/uv project** (plain requirements.txt + `pytest` on
  PATH assumed by its makefile). Local runs: `uv run --python 3.11
  --with-requirements <reqs minus psycopg2> --with pytest --with pytest-asyncio`
  (psycopg2 needs pg_config on macOS; tests never connect — use a lazy
  `postgresql+asyncpg://` DATABASE_URL, NOT sqlite: banking models use the Postgres
  UUID dialect).
- `saga.id = correlation_id` in the orchestrator (verified in
  `orchestrator.start_saga`) — the `saga_id` in saga commands IS the id banking
  minted, so the consumer can scope its claim-release to it.
- No DB-level repo tests possible cheaply here (no sqlite compat, no testcontainers) —
  claim SQL semantics verified live instead; noted as accepted gap.
