# Session 2026-07-13 — AI-20 gates: decision + eval hardening

Closed the two pre-code gates for AI-20 (ES hybrid search replaces ChromaDB), per
[plans/2026-07-12-ai-service-es-chat.md](../plans/2026-07-12-ai-service-es-chat.md).

## Done

- **Gate 1 — embed-worker placement decision** recorded in
  [decisions/2026-07-13-embed-worker-placement.md](../decisions/2026-07-13-embed-worker-placement.md):
  separate consumer inside analytics-service on its own queue
  (`analytics.embeddings`, own DLQ, bound to `transaction.*`), partial-updating only
  `description_vector`. Rejected: embedding in the main projector (couples core
  projections to Ollama uptime), ai-service writing ES directly (breaks ADR-0004
  ownership), batch-only (reintroduces P3-04 staleness). Key derived requirement:
  the hybrid endpoint must degrade to BM25-only when the vector is missing.
- **Gate 2 — eval hardening** (`services/ai-service/tests/eval/`):
  - 22 distractor transactions (ids 100+): semantically adjacent to golden queries
    (abonnement-not-streaming: Telia/YouSee; bil-not-benzin: AutoMester/FDM;
    sundhed-not-apotek: tandlæge/optiker; kiosk, biograf, forsikring, gaver, hjem &
    indretning, personlig pleje). Real Danish orthography (æøå) vs. the
    ASCII-transliterated original corpus → mixed-spelling retrieval is exercised.
    Distractor categories deliberately lack `CATEGORY_SYNONYMS` entries; conversely
    Underholdning's synonym text ("streaming, abonnement, film") makes biograf/Tivoli
    docs naturally hard distractors for "streaming abonnementer".
  - 15 new hard retrieval cases (cross-spelling "Føtex" vs. doc "Foetex",
    world-knowledge "musik"→Spotify, "biograf film" where streaming docs literally
    contain "film", near-distractor discrimination like "bilforsikring" vs.
    FDM/AutoMester/Tryg) + 6 harder intent phrasings.
  - **recall@3 added as the sharp metric** — recall@10 stays saturated over the ~67-doc
    corpus; top-3 ranking is where hybrid vs. pure-vector will differ. Same search call,
    two cuts — no extra runtime.
  - 4 aggregation literals updated (distractors changed April/May totals); the
    no-Ollama drift-guard selfcheck pins all 50 golden↔fixture relations.

## Measured baseline (ChromaDB + bge-m3, compose-ollama 11435)

| metric | 2026-07-12 (saturated) | 2026-07-13 (hardened) |
|---|---|---|
| retrieval recall@10 | 1.000 (20 cases) | 1.000 (35 cases — still saturated) |
| retrieval recall@3 | — | **0.967** |
| retrieval MRR | 1.000 | **0.981** |
| intent accuracy | 1.000 (16) | **0.955** (22) |

Signal carriers: "tøj shopping" (recall@3 0.33, RR 0.33 — distractors outrank the
relevant docs), "el og vand regninger" (recall@3 0.50 — forsikring/telefon bills invade
top-3), intent miss on "Hvor stor en bid tager boligudgifterne af mit samlede forbrug?"
(→ budget_status, expected category_breakdown). **The AI-20 cutover gate is now
recall@3 + MRR, not recall@10.** Floors: recall@10/recall@3/MRR = 0.95, intent 0.90.

## Learned / gotchas

- This Mac has host-Ollama (11434) with **qwen3:8b** + bge-m3 — it is the machine with
  the responder model; the pending full SSE-pipeline smoke can run HERE.
  Compose-ollama (11435) has bge-m3 + qwen3:4b.
- Eval runtime is memory-pressure-bound, not corpus-bound: full run took 37:46 with
  qwen3:4b loaded/swapping on the docker VM, but retrieval-only rerun took 2:42 over
  the same corpus. Don't read runtime as a regression signal.
- Distractor design has to respect the aggregation cases: category choice (e.g. IKEA →
  "Hjem & indretning", not "Bolig"; Taxa in May only) keeps most literals stable —
  only 4 needed updating, verified by the selfcheck.
