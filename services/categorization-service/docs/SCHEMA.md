# Categorization Service — Database Schema Design

**Status:** Pre-migration design document (all questions resolved)
**Date:** 2026-04-23
**Update policy:** Mark as "as-built" when migrations are complete. Update when schema changes.

## Context

Categorization-service owns the taxonomy (categories, subcategories, merchants),
the rule engine, and the categorization pipeline (rules, ML, LLM).
It is extracted from the monolith as part of the strangler fig migration.

The service exposes two inbound interfaces:

- `POST /api/v1/categorize/` — sync tier 1 (rule engine), called by transaction-service
- `transaction.created` event consumer — async full pipeline (tier 1-3)

It emits:

- `transaction.categorized` events (categorization results)
- `category.*` events (taxonomy changes)

## Table Overview

| Table | Role | Master/Denorm | Row growth |
|-------|------|---------------|------------|
| `categories` | Top-level taxonomy | Master | Static (~10 rows) |
| `subcategories` | Second-level taxonomy | Master | Static (~35 rows) |
| `merchants` | Learned merchant entities | Master | Slow (grows with new merchants) |
| `categorization_rules` | Data-driven matching rules | Master | Slow (seed + user rules) |
| `categorization_results` | Audit trail of decisions | Master | Linear with transactions (append-only) |
| `outbox_events` | Transactional outbox | Infrastructure | Transient (published rows cleaned) |
| `processed_events` | Consumer inbox (dedup) | Infrastructure | Transient (old rows cleaned) |

## Schema Definitions

### categories

```sql
CREATE TABLE categories (
    id          INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    name        VARCHAR(45) NOT NULL UNIQUE,
    type        VARCHAR(20) NOT NULL,       -- 'expense' | 'income' | 'transfer'
    display_order INTEGER NOT NULL DEFAULT 0,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ
);

CREATE INDEX ix_categories_name ON categories (name);
```

**Seed:** IDs 1-10 are pinned to match monolith MySQL and transaction-service's
`005_seed_default_categories` migration. After seed, sequence is reset to
`MAX(id)` so new rows start at 11+.

**Delete policy:** Hard delete with guard — reject if any `subcategories` rows
reference the category (`ON DELETE RESTRICT` on the FK from subcategories).

### subcategories

```sql
CREATE TABLE subcategories (
    id          INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    name        VARCHAR(100) NOT NULL,
    category_id INTEGER NOT NULL REFERENCES categories(id) ON DELETE RESTRICT,
    is_default  BOOLEAN NOT NULL DEFAULT TRUE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX ix_subcategories_category_id ON subcategories (category_id);
CREATE INDEX ix_subcategories_name ON subcategories (name);
```

**Seed:** IDs are dumped from monolith MySQL `SubCategory` table and inserted with
explicit IDs to preserve cross-service compatibility. Sequence reset after seed.

**Delete policy:** Hard delete with guard — reject if any `categorization_rules`
reference the subcategory (enforced by FK `ON DELETE RESTRICT`). Application-level
guard also checks if any `categorization_results` reference it.

### merchants

```sql
CREATE TABLE merchants (
    id                INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    normalized_name   VARCHAR(200) NOT NULL UNIQUE,
    display_name      VARCHAR(200) NOT NULL,
    subcategory_id    INTEGER NOT NULL REFERENCES subcategories(id) ON DELETE RESTRICT,
    transaction_count INTEGER NOT NULL DEFAULT 0,
    is_user_confirmed BOOLEAN NOT NULL DEFAULT FALSE,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX ix_merchants_normalized_name ON merchants (normalized_name);
CREATE INDEX ix_merchants_subcategory_id ON merchants (subcategory_id);
```

**Seed:** Populated from `SEED_MERCHANT_MAPPINGS` in taxonomy.py. Each keyword
becomes a merchant row with the normalized keyword as `normalized_name`.

**Delete policy:** No delete method. Merchants are learned entities that accumulate.

### categorization_rules

```sql
CREATE TABLE categorization_rules (
    id                      INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    user_id                 INTEGER,        -- NULL = system rule, else user-specific
    priority                INTEGER NOT NULL,
    pattern_type            VARCHAR(30) NOT NULL,  -- 'keyword' | 'regex' | 'merchant' | 'amount_range'
    pattern_value           TEXT NOT NULL,
    matches_subcategory_id  INTEGER NOT NULL REFERENCES subcategories(id) ON DELETE RESTRICT,
    active                  BOOLEAN NOT NULL DEFAULT TRUE,
    created_at              TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at              TIMESTAMPTZ
);

CREATE INDEX ix_rules_active_priority ON categorization_rules (active, priority);
CREATE INDEX ix_rules_user ON categorization_rules (user_id) WHERE user_id IS NOT NULL;
```

**Design rationale — rules as data, not code:**

The monolith stores rules in `taxonomy.py` as a Python dict. Moving to a database
table enables: (a) audit trail on rule changes, (b) runtime updates without redeploy,
(c) user-specific rules that override system defaults, (d) A/B testing of rule sets
via `model_version` on results.

**Priority scheme:** System rules are seeded at priority intervals of 100 (100, 200,
300...) to leave gaps for user rules. User rules default to priority 50 (higher
priority = wins first in evaluation). Within the same priority, the rule engine
applies longest-match-first as a secondary sort — this preserves the monolith's
implicit ordering behavior.

**Rule evaluation:** `SELECT * FROM categorization_rules WHERE active = TRUE AND
(user_id = :uid OR user_id IS NULL) ORDER BY priority ASC, length(pattern_value) DESC`.
First match wins.

**Seed:** Each entry in `SEED_MERCHANT_MAPPINGS` becomes one row with
`pattern_type='keyword'`, `user_id=NULL`, priority assigned in 100-intervals
per subcategory group.

### categorization_results

```sql
CREATE TABLE categorization_results (
    id              INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    transaction_id  INTEGER NOT NULL,
    category_id     INTEGER NOT NULL,
    subcategory_id  INTEGER NOT NULL,
    merchant_id     INTEGER,
    tier            VARCHAR(20) NOT NULL,   -- 'rule' | 'ml' | 'llm' | 'manual' | 'fallback'
    confidence      VARCHAR(10) NOT NULL,   -- 'high' | 'medium' | 'low'
    model_version   VARCHAR(50) NOT NULL,   -- e.g. 'rules-v1', 'ml-v2.3'
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX ix_results_transaction_latest
    ON categorization_results (transaction_id, created_at DESC);
```

**Design:** Append-only audit trail. One row per categorization attempt, not per
transaction. A transaction re-categorized by tier 2/3 gets additional rows —
latest row (by `created_at DESC`) is the current truth.

**No FK to categories/subcategories:** Intentional. Historical results must survive
taxonomy changes. If a subcategory is renamed or restructured, old results remain
intact with the IDs that were valid at categorization time.

**No partitioning at launch:** Postgres handles millions of rows efficiently with
the compound index on `(transaction_id, created_at DESC)`. Partitioning by month
is a future optimization if the table exceeds ~50M rows.

### outbox_events

```sql
CREATE TABLE outbox_events (
    id              VARCHAR(36) PRIMARY KEY,    -- UUID string
    aggregate_type  VARCHAR(50) NOT NULL,
    aggregate_id    VARCHAR(100) NOT NULL,
    event_type      VARCHAR(100) NOT NULL,
    payload_json    TEXT NOT NULL,
    correlation_id  VARCHAR(36),
    status          VARCHAR(20) NOT NULL DEFAULT 'pending',
    attempts        INTEGER NOT NULL DEFAULT 0,
    next_attempt_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    published_at    TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX ix_outbox_pending_poll
    ON outbox_events (status, next_attempt_at, created_at);
```

**Identical to user-service and transaction-service outbox schemas.** This is the
established standard across the platform. Do not deviate.

### processed_events (inbox)

```sql
CREATE TABLE processed_events (
    id              INTEGER PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    message_id      VARCHAR(36) NOT NULL,
    consumer_name   VARCHAR(100) NOT NULL,
    event_type      VARCHAR(100) NOT NULL,
    processed_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (message_id, consumer_name)
);

CREATE INDEX ix_processed_at ON processed_events (processed_at);
```

**Naming note:** The monolith's inbox uses `correlation_id` for this column. In
`BaseEvent`, `correlation_id` is generated as `uuid4()` per event (unique per
message by default), but callers can override it to a shared trace ID. The name
`message_id` makes the intent explicit: this column deduplicates individual
messages, not conversation chains. When consuming events, map
`event.correlation_id` to `processed_events.message_id`.

**Cleanup:** Old rows (>7 days) are purged periodically by the consumer process
at startup, matching the monolith's `BaseConsumer._cleanup_old_events` pattern.

## Cross-Cutting Design Decisions

### Primary keys: Integer, not UUID

**Decision:** All domain entity PKs use `INTEGER` with auto-increment.

**Rationale:** Every existing service (user-service, transaction-service, monolith
MySQL) uses integer PKs. Categories 1-10 are pinned across services. UUID would
break cross-service ID compatibility and require a mapping layer during migration.

**Trade-off acknowledged:** Integer PKs can collide if data is merged from multiple
sources. Not relevant for this single-tenant application. If multi-tenancy is
introduced, reassess.

### ID preservation from monolith

**Decision:** Seed categories and subcategories with their existing monolith MySQL IDs.

**How:** Alembic `op.execute()` with `INSERT INTO ... (id, ...)` using explicit IDs,
followed by `SELECT setval('tablename_id_seq', (SELECT MAX(id) FROM tablename))`.

**Why:** Eliminates the need for an ID mapping table. Transaction-service's
`transactions.category_id` and `transactions.subcategory_id` already reference these
integer IDs. Preserving them means zero data migration on the transaction side.

### Delete policy: hard delete with FK RESTRICT guards

**Decision:** No soft delete. Hard delete with database-level and application-level
guards.

| Entity | Guard | Enforcement |
|--------|-------|-------------|
| Category | Has subcategories? | FK `ON DELETE RESTRICT` on `subcategories.category_id` |
| SubCategory | Has active rules? | FK `ON DELETE RESTRICT` on `categorization_rules.matches_subcategory_id` |
| SubCategory | Has results? | Application-level check before delete |
| Merchant | No delete | No delete method exposed |
| Rules | Direct delete | No guard needed — rules are leaf nodes |
| Results | No delete | Append-only, never deleted |

**Rationale:** Matches the monolith's existing behavior. Soft delete adds complexity
(filtered queries everywhere, `deleted_at` checks) without a clear use case. The
FK RESTRICT guards prevent orphaned references, which is the actual risk.

### Seed strategy: migration for baseline, script for iteration

**Decision:** Initial taxonomy data is seeded in Alembic migrations (deterministic,
version-controlled, runs automatically in CI). Future keyword additions use either
new migrations or an admin API.

**Advantage:** `alembic upgrade head` on a fresh database produces a fully functional
service with all categories, subcategories, merchants, and rules. No manual steps.

### categorization_rules priority intervals

**Decision:** System rules seeded at priority 100, 200, 300... User rules default
to priority 50.

**Rationale:** Lower priority number = evaluated first (higher precedence). User
rules at 50 beat system rules at 100+. Gaps of 100 between system rules allow
inserting new system rules (e.g., 150) without renumbering. Matches the YNAB/
Lunchmoney pattern where user rules override system defaults.

## Resolved Questions

### 1. Subcategory IDs: preserved from monolith MySQL

**Decision:** Dump monolith `SubCategory` table and seed with explicit IDs.

Subcategory IDs are referenced by `transactions.subcategory_id` across services.
Generating new IDs would require an ID mapping table that lives forever and
complicates every cross-service query and debugging session. Preserving IDs
eliminates that accidental complexity entirely.

After seed INSERT, `setval('subcategories_id_seq', MAX(id))` must be called
to prevent auto-increment collision when new subcategories are created later.
An integration test verifies this: create a subcategory via the repository and
assert `id > MAX(pinned_ids)`.

### 2. Merchant IDs: auto-generated (not preserved)

**Decision:** Let Postgres auto-generate merchant IDs from `SEED_MERCHANT_MAPPINGS`.

No service outside categorization-service references `merchants.id` as a FK.
Transactions point to `subcategory_id`, not `merchant_id`. Merchants are an
internal lookup table for the rule engine. Preserving monolith IDs would add
complexity without benefit.

Validation: confirmed via grep that `merchant_id` is only used internally in
`categorization_results.merchant_id` (cat-service-owned) and never in
transaction-service or monolith transaction tables.

### 3. model_version naming: `<tier>-<implementation>-v<semver>`

**Decision:** Structured format that is parseable and human-readable.

Format: `split("-", 2)` yields `[tier, implementation, version]`.

| Version string | Meaning |
|----------------|---------|
| `rules-keyword-v1` | Tier 1, keyword matching, initial rule set |
| `rules-keyword-v2` | Tier 1, algorithm change (e.g. matching logic) |
| `ml-fasttext-v1` | Tier 2, fastText classifier, first trained model |
| `ml-bert-danish-v1` | Tier 2, new architecture |
| `llm-gpt4-v1` | Tier 3, GPT-4 with initial prompt |
| `llm-llama3-local-v1` | Tier 3, local LLM |

**When to bump:** Whenever observable output can change. Editing a keyword in
`categorization_rules` is a data change (no bump). Changing the rule engine's
matching algorithm is a code change (bump). Retraining ML on new data is a
model change (bump — embeddings can drift).

**Active version tracking:** Configuration (env var or DB table), not hardcoded.
Enables rollback without deploy.
